## 3. 技术架构（一张图）

```
 📱 Android App                🖥️ 你的后端(公网)            🤖 大模型
 ┌─────────────────┐  fetch   ┌──────────────────┐  调用   ┌────────────┐
 │ WebView 里跑     │ ───────▶ │ FastAPI 代理      │ ──────▶ │ DeepSeek   │
 │ 现有 HTML 原型   │ ◀─────── │ (握着 API key)    │ ◀────── │            │
 └─────────────────┘  JSON    └──────────────────┘         └────────────┘
   Capacitor 套壳                key 只在这里,不进 apk
```

**关键安全红线**: API key **永远只在后端**,不进 apk、不进 git、不进提交包。app 通过调后端间接使用大模型。

## 5. 核心链路（P0 —— 唯一要"真跑"的那条）

> 演示的灵魂。其余全部用现有 mock / 录屏。

**"说一句话 → 真生成带依赖关系的日程"**

```
用户说一句话("明天上午写实验,晚上八点陪妈妈散步,这周去趟医院")
  → 前端把这句话发给后端 /api/parse
  → 后端把它 + 提示词发给 DeepSeek
  → DeepSeek 返回规范 JSON(切分成3件事、各自领域/时长/时间)
  → 前端用这个真 JSON 渲染成任务卡片(渲染代码已现成,不用改)
```

**落点在现有代码里**:现在这一步是本地函数 `parseUtterance()` 用关键词假装解析;**把它改成 fetch 后端**即可。下游渲染完全复用。

## 10. 提交物 & "评委一定看得到效果"的兜底

**提交物**(暂定):
- **apk 安装包**(评委可直接装到安卓手机)
- **后端部署 URL**(演示时后端必须活着,app 才能"真跑")
- **源码 zip**(前端 + 后端,**不含 API key**;附 `.env.example` 占位)
- **演示视频**(3-5 分钟)

**核心兜底(重要)**: 现有 HTML 本来就是全 mock。让核心链路那个函数写成
`try{ 调 DeepSeek }catch{ 退回本地 parseUtterance 假数据 }`,
这样**后端万一挂了 / 评委没网 / DeepSeek 限流,也不会白屏**,顶多退回演示版。
再加**录屏视频**做终极保险。→ 无论如何评委都能看到效果。
